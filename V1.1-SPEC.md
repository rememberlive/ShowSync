- **Persistence is an explicit field map** (`Config.swift:336–366`): adding a

## 2. The platform flag

### 2.1 New runtime field (additive, not persisted)

`Config.swift` — one line added to `struct Config`:

```swift
// V1.1 Windows-target path — UNTESTED against live Windows Backup as of this commit (Windows sshd pending).
var backupPlatform: String = ""   // "" = mac (today's behavior); "windows" = ShowSync-Win Backup
```

Not added to `MainConfig`/`save()` → never written to disk → no migration, no
schema change. (Its `didSet` save-churn is avoided by write-on-change guards at
every assignment site, same idiom the file already uses.)

### 2.2 TXT parse (automatic mode)

`BonjourDiscovery.swift`:

- `DiscoveredBackup` (`:211`): add `var platform: String = ""` (defaulted —
  existing initializers and derived `Equatable` unaffected).
- `handleResolverEvent` (`:395` block): add one parse line beside the existing ones:
  `let platform = txt["platform"] ?? ""` and pass it into the `DiscoveredBackup` init.
- `handleAutoReconnect` (`:481`, inside the existing matched-peer block): guarded
  write-on-change, **always assigned** (absent key → `""`), so a stale `windows`
  flag self-heals the moment a Mac Backup is adopted:

```swift
if ConfigStore.shared.config.backupPlatform != backup.platform {
    ConfigStore.shared.config.backupPlatform = backup.platform
}
```

The other additive Win TXT keys (`role`, `ver`) stay ignored in V1.1.

### 2.3 Manual mode (task item 3) — proposal + one open question

Manual mode has no TXT. Today's manual poll runs
`cat "$HOME/Library/Application Support/Sync/config_backup.json"` — on Windows
sshd (default shell cmd.exe) `cat`/`$HOME` don't exist, the command exits non-zero,
and `ConnectionStatus` would show **unreachable** even though SSH works.

**Proposed (explicit, no heuristics): a user-set "Windows Backup" toggle**, shown
only in manual mode's Destination section (`SettingsView.swift` manual branch,
`:1049`):

- Toggle ON → `config.backupPlatform = "windows"` and the Folder row becomes an
  editable text field for a Windows path (`C:/Users/<user>/Sync`; backslashes
  normalized to `/` on save). Stored in the existing `config.backupDestination`
  (runtime field, already not persisted for Main).
- When `backupPlatform == "windows"`, `ConnectionStatus.check()` gets a gated
  branch that uses the **existing auto-mode probe** (`ssh … exit`) instead of the
  `cat` pipeline (insert-only `if` at the top of the manual branch, `:98`); the
  Windows transport's own preflight supplies free-space/verify-request instead.
- `confirmBackupDestination` (`SettingsView.swift:2264`) gains the same gate: for
  Windows targets it runs the new module's write-test probe instead of the
  `cat config_backup.json` read.
- Toggle OFF (default) → everything exactly as today, including for existing users
  (flag defaults `""`).

> **Open question for approval:** ShowSync-Win — does it write a
> `config_backup.json` (and where), and is there a remote command you'd rather I
> read it with? If yes I can auto-fill dest/platform in manual mode too; the
> toggle design above assumes we *don't* rely on that and let the user enter the
> path. Also: does the Win engine consume the signal files
> (`.sync_start/.sync_progress/.sync_complete`) and produce `.sync_refused` /
> `.verify_request`? The design writes/reads them portably (via sftp) on the
> assumption it does; if not, those calls are cheap no-ops.

## 4. New module — `Sync/WindowsTransport.swift` (one new file)

Every declaration headed by:
`// V1.1 Windows-target path — UNTESTED against live Windows Backup as of this commit (Windows sshd pending).`

### 4.1 Principles

- **No rsync.** Push via `/usr/bin/sftp` batch (`put -p` = mtime-preserving);
  `scp -p` kept as documented fallback for single files if sftp batch misbehaves.
- **No POSIX remote shell assumptions.** Remote queries run PowerShell via
  `ssh … powershell.exe -NoProfile -NonInteractive -EncodedCommand <base64-UTF16LE>`
  — immune to cmd-vs-PowerShell default-shell quoting.
- **Remote paths use forward slashes** (`C:/Users/<user>/Sync/...`). A single
  normalizer (`\` → `/`, strip trailing `/`) is applied to everything entering the
  module. Empty destination → `"Sync"` (relative to sshd home — the Windows
  analogue of the `~/Sync` fallback).
- **One-way, additive, never deletes remote files.** The batch contains only
  `-mkdir` (dash prefix = ignore already-exists) and `put -p`. No `rm` of user
  files, ever. (Protocol signal files are the only remote writes besides `put`.)
- Same network discipline: `-o BatchMode=yes -o ConnectTimeout=…
  -o StrictHostKeyChecking=no` and `-o BindAddress=<bindIP>` when a preferred
  interface is set (sftp/scp pass `-o` through to ssh — same trust, same NIC pinning).
- **Exclusion parity:** local enumeration filters with `fnmatch` against
  `RsyncExclusions.patterns` (the shared source of truth) so sync and verify
  compare the same file set as the Mac path.

### 4.2 Sync algorithm (`windowsSync`)

1. **Preflight/write-test** — sftp batch: `-mkdir "<dest>"`, `put <tmp> "<dest>/.sync_writetest_<n>"`, `rm` it.
   Unwritable → fall back to `"Sync"` + the existing `fallbackNotice` text; ssh
   error → proceed (mirrors `.testFailed` semantics).
2. **Refused check** — sftp batch `ls "<dest>/.sync_refused"`; exit 0 → surface the
   existing low-space notice + `refused` log entry (mirrors `checkSyncRefused` timing: also re-checked after transfer).
3. **Remote manifest** — PowerShell:
   `Get-ChildItem -LiteralPath <dest> -Recurse -File` →
   `<unix-mtime>\t<bytes>\t<relpath>` lines (UTF-8). Missing dir → empty manifest.
4. **Local manifest** — `FileManager` enumerator over the source, exclusions applied.
5. **Diff (additive)** — upload set = files absent remotely, or size differs, or
   mtime differs beyond tolerance (**±2 s**, absorbing sftp's whole-second
   SETSTAT truncation vs NTFS 100 ns stamps). Exact `expectedSize` for the
   progress bar comes from this diff — no estimate dry-run needed.
6. **Signal files** — local temp JSONs pushed via sftp: `.sync_start` before,
   `.sync_progress` during (throttled rewrites), `.sync_complete` + removal of
   start/progress after (same JSON payloads as `writeSyncStart/Progress/Complete`).
7. **Transfer** — one sftp `-b` batch: depth-sorted `-mkdir` lines, then
   `put -p "<local>" "<dest>/<rel>"` per file. Output streamed; each
   `Uploading <x>` line advances per-file results + `SyncProgress` (bytes summed
   from the manifest — replaces du-polling, which is skipped for Windows).
8. **Completion** — same state machine: `.done`/`.error("Sync interrupted — …")`,
   `lastSyncTime`, icon state, `TransferLogEntry` (real per-file counts/bytes from
   the batch results), 3 s revert to `.ready`. Cancel: terminate the sftp Process
   (task-slot shared via the same `task`-style private storage in the extension),
   clean signal files — surfaced identically.
9. **Version history / prune** — **not supported on Windows targets in V1.1**
   (POSIX `cp`/`find` protocol). Skipped with an `NSLog`; spec'd for V1.2.
   "Check Files" preview: implemented from the same diff (steps 3–5) with the
   existing `DryRunResult` formatting — no rsync.

### 4.3 Verify algorithm (`windowsVerify`)

- **Fast** (`config.fastVerify == true`): remote manifest (4.2 step 3) vs local
  manifest — size + mtime ±2 s, source-side only (extra remote files ignored,
  matching push-direction rsync semantics). Differ count → existing
  `.verified(deep: false)` / `.differs(n)` statuses.
- **Deep** (default): remote PowerShell
  `Get-ChildItem -Recurse -File | Get-FileHash -Algorithm SHA256` →
  `<hash>\t<relpath>`; local SHA256 via CryptoKit streaming (1 MB chunks, off-main).
  Missing-remote or hash-mismatch → differ count. Statuses/timeouts/10 s
  auto-clear identical; `isRemoteVerify` result write (`ok/differs:n/error`) goes
  through an sftp `put` of `.verify_result` + `rm` of `.verify_request`.
- Cancel verify: same `cancelVerify()` surface; the extension registers its
  Process in `verifyTask`-equivalent storage so the existing button works.

## 5. Proof the no-flag path is untouched

1. **Gate value:** `backupPlatform` defaults `""`; the only writes are
   TXT-adoption (`txt["platform"] ?? ""`), the manual-mode toggle, and JSON parse
   sites — a Mac Backup can never produce `"windows"`, and absent-key assignment
   actively resets stale flags.
2. **Insert-only diffs:** both engine gates are `if … { …; return }` inserted
   between existing statements; `git diff` on `MainView.swift` will show only
   added lines — no existing line modified, reordered, or wrapped.
3. **Persistence:** `save()`'s explicit `MainConfig(...)` field list is not
   edited → `config_main.json` bytes unchanged; `backupPlatform` is runtime-only.
4. **Discovery:** `DiscoveredBackup.platform` is a defaulted field parsed from a
   key Mac Backups never send; `handleAutoReconnect`'s new assignment is guarded
   write-on-change and assigns `""` for Mac peers (no observer churn, no
   behavioral delta).
5. **UI:** the manual-mode toggle renders only in manual mode and defaults off;
   all other views read the same `@Published` state they read today.
6. **New code is unreachable** with the flag empty: `WindowsTransport.swift` has
   no callers outside the two gates and the toggle.
7. Verified by Phase 3: clean build + the two-Mac regression checklist (discover,
   pair, sync, verify fast+deep, rename, forget) against your Mac Backup.

## 8. Approval record (§7 decisions — 2026-07-05)

1. **Manual mode: APPROVED as designed** — explicit "Windows Backup" toggle +
   editable Windows path. Do NOT read ShowSync-Win's config remotely in V1.1 (no
   coupling to the Windows app's internal file layout; its config format freezes
   when the Windows Backup role is productized — remote auto-fill is a V1.2
   candidate then).
2. **Signal files: YES by specification** — ShowSync-Win's engine was transcribed
   1:1 from this Mac source (identical signal-file names, JSON payloads, 0.5 s
   Backup signal poll): it consumes `.sync_start/.sync_progress/.sync_complete`
   and produces `.sync_refused`/`.verify_request` at the destination root.
   **Caveat:** the Windows Backup receive role is spec-complete but not yet
   hardware-exercised (Windows sshd pending) — spec-confirmed, hardware-unproven,
   consistent with the UNTESTED markers; the deferred end-to-end test validates it.
3. **mtime tolerance ±2 s: CONFIRMED** — matches ShowSync-Win exactly (UTC epoch
   seconds, ±2 s documented slack; hardware-measured delta was 0 s).
4. **Preview ("Check Files") via manifest diff: INCLUDED** — same-comparator
   principle, keeps UI parity.

Also approved as spec'd: version history/prune **skipped on Windows targets**
with a logged skip (V1.2 item); **StrictHostKeyChecking stays Mac-native
semantics** (`StrictHostKeyChecking=no`, same option set as the existing engine —
do not import the Windows accept-new decision; minimum change to this codebase).

### 8.1 Implementation addenda (Phase 2, disclosed deltas from §3)

- **Cancel paths need the same gate.** `cancel()` and `cancelVerify()`
  (`MainView.swift:621/:801`) read private per-run metadata (`task`,
  `isAutoSync`, `syncRemotePath`, …) that a Windows run never populates — without
  a gate, Cancel would leave the sftp child running and log a wrong entry. Each
  gets the identical insert-only pattern:
  `if WindowsTransport.shared.isSyncActive { WindowsTransport.shared.cancelSync(); return }`
  (resp. `isVerifyActive`/`cancelVerify`). No-flag path: `isSyncActive` is false
  whenever no Windows transfer runs → fall-through byte-identical.
- **One access-level widening:** `SyncEngine.isRemoteVerify`
  (`MainView.swift:149`) goes `private var` → `var` so the Windows verify path
  (separate file) can honor Backup-requested verifies and write `.verify_result`
  exactly like the mac path. Behavior-identical; no logic touched.
- The `SettingsView` manual Folder row is gated by adding
  `, store.config.backupPlatform != "windows"` to its existing condition
  (one-line, no-flag-path-identical) with a new sibling `== "windows"` block.
  The `ConnectionStatus` manual poll needed NO condition edit as-built: the
  Windows branch is an insert-only early-exit above the existing `if isManualMode`.
- **As-built audit (git diff, non-insert lines):** exactly three existing lines
  changed across all Swift files — (1) the `isRemoteVerify` access widening above;
  (2) the SettingsView Folder-row condition above; (3) the `DiscoveredBackup`
  initializer call gained a trailing `platform:` argument (value `""` for every
  Mac peer — behavior-identical). Everything else is pure insertion plus the new
  `WindowsTransport.swift` and the four project-file registration lines.
